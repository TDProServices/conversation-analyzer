### Issue #5: Confidence scoring for extracted items

**Labels:** `enhancement`, `feature`, `priority:high`, `phase:5`

**Description:**
Rate confidence in each extracted TODO/bug/feature (0-100%) to help user prioritize review.

**Use Cases:**
- Explicit TODO marker (`TODO:`) = 95-100% confidence
- "I need to..." = 80-90% confidence
- "We should probably..." = 60-75% confidence
- Vague mention = 40-60% confidence

**Implementation Notes:**
- Explicit markers (TODO:, FIXME:, BUG:) = high confidence
- Imperative sentences ("I will...", "I need to...") = medium-high confidence
- Conditional statements ("should", "could", "might") = medium confidence
- Questions about issues = medium-low confidence
- LLM can help score ambiguous cases

**Output:**
```markdown
### High Confidence TODOs (90%+)
1. [95%] Fix authentication bug (TODO: explicitly marked)
2. [92%] Add tests for parser ("I need to add tests" - clear commitment)

### Medium Confidence TODOs (60-89%)
1. [75%] Consider caching ("we should probably cache results")
2. [68%] Improve error messages ("could be better")
```

**Priority:** High (helps users trust the tool)

**Phase:** 5 (needs core extraction working first)

### Issue #7: Deduplication across conversations

**Labels:** `enhancement`, `feature`, `priority:high`, `phase:5`

**Description:**
Avoid reporting the same TODO multiple times when it appears in different conversations or files.

**Scenario:**
- Conversation 1 (2025-11-15): "I need to fix the authentication bug"
- Conversation 2 (2025-11-16): "Still need to fix that auth bug"
- Conversation 3 (2025-11-17): "TODO: Fix authentication issue"

**Expected:**
Report as ONE item with:
- First mentioned: 2025-11-15
- Last mentioned: 2025-11-17
- Frequency: 3 occurrences
- Confidence: High (multiple mentions + explicit TODO marker)

**Algorithm:**
1. Exact match: Same text = duplicate
2. Fuzzy match: >80% similar = likely duplicate
3. Semantic match: LLM determines if same intent = duplicate

**Configuration** (.conversation-analyzer.yaml):
```yaml
deduplication:
  similarity_threshold: 80  # percent
  time_window_days: 7       # dedupe within window
  strategy: "semantic"      # exact, fuzzy, or semantic
```

**Priority:** High (prevents noise in reports)

**Phase:** 5 (after basic extraction works)

**Related:** Issue #5 (confidence scoring)

### Issue #8: Priority scoring for extracted items

**Labels:** `enhancement`, `feature`, `priority:high`, `phase:5`

**Description:**
Automatically score priority of extracted items based on context, urgency keywords, and impact.

**Scoring Factors:**
1. **Urgency keywords (40% weight):**
   - "critical", "urgent", "asap", "blocking" ‚Üí High priority
   - "soon", "important" ‚Üí Medium priority
   - "eventually", "someday" ‚Üí Low priority

2. **Impact keywords (30% weight):**
   - "crash", "broken", "data loss", "security" ‚Üí High impact
   - "slow", "confusing", "annoying" ‚Üí Medium impact
   - "nice to have", "polish" ‚Üí Low impact

3. **Frequency (20% weight):**
   - Mentioned 5+ times ‚Üí High priority
   - Mentioned 2-4 times ‚Üí Medium priority
   - Mentioned once ‚Üí Priority depends on keywords

4. **Recency (10% weight):**
   - Last 7 days ‚Üí boost priority
   - Last 30 days ‚Üí neutral
   - Older ‚Üí reduce priority

**Output:**
```markdown
## üî¥ High Priority (8 items)
1. [CRITICAL] Fix authentication crash - mentioned 3x, last: 2 days ago
   Source: auth-bug.md:42, conversation-2025-11-15.md:156

## üü° Medium Priority (12 items)
...

## üü¢ Low Priority (5 items)
...
```

**Configuration** (.conversation-analyzer.yaml):
```yaml
priority:
  weights:
    urgency: 0.4
    impact: 0.3
    frequency: 0.2
    recency: 0.1
  urgency_keywords:
    - critical
    - urgent
    - asap
```

**Priority:** High (helps users focus on important items)

**Phase:** 5
